Approach 1
Docusaurus postBuild Plugin + Nginx Content-Type
New plugin copies source .md/.mdx files into the build output; nginx serves them as plaintext.
Low Complexity Recommended

How it Works

A new Docusaurus plugin hooks into the postBuild lifecycle. After Docusaurus finishes building the HTML site into build/, the plugin walks the docs/ source tree and copies every .md/.mdx file into the equivalent path under build/docs/. Since nginx's try_files $uri already tries the exact URI first, a request for /docs/foo/bar.md will resolve to the copied file. A single nginx location block adds the text/plain content-type header so browsers display raw text instead of prompting a download.

+import fs from 'fs';
+import path from 'path';
+
+import type { LoadContext, Plugin } from "@docusaurus/types";
+
+export default async function mdExportPlugin(
+  context: LoadContext,
+): Promise<Plugin> {
+  return {
+    name: 'stellar-docs-md-export-plugin',
+
+    async postBuild({ outDir }) {
+      const docsSourceDir = path.join(context.siteDir, 'docs');
+      const docsOutputDir = path.join(outDir, 'docs');
+      let count = 0;
+
+      copyMdFiles(docsSourceDir, docsOutputDir, () => count++);
+      console.log(`✅ md-export: copied ${count} source files to build output`);
+    },
+  };
+}
+
+function copyMdFiles(
+  sourceDir: string,
+  destDir: string,
+  onCopy: () => void,
+): void {
+  if (!fs.existsSync(sourceDir)) return;
+  const entries = fs.readdirSync(sourceDir, { withFileTypes: true });
+
+  for (const entry of entries) {
+    const srcPath = path.join(sourceDir, entry.name);
+    const dstPath = path.join(destDir, entry.name);
+
+    if (entry.isDirectory()) {
+      copyMdFiles(srcPath, dstPath, onCopy);
+    } else if (/\.(md|mdx)$/.test(entry.name)) {
+      fs.mkdirSync(path.dirname(dstPath), { recursive: true });
+      fs.copyFileSync(srcPath, dstPath);
+      onCopy();
+    }
+  }
+}

Approach 2
npm postbuild Script
A standalone Node.js script runs after yarn build, copying source Markdown files into the build output directory.
Low Complexity

How it Works

npm's postbuild lifecycle hook automatically runs a script after yarn build completes. The script walks the docs/ source tree and copies .md/.mdx files to the matching paths in build/docs/. This approach is framework-agnostic: it doesn't depend on Docusaurus internals and could survive a Docusaurus major-version upgrade. The nginx change is identical to Approach 1.

Approach 3
Nginx Redirect → GitHub Raw CDN
A single nginx location block issues a 302 redirect, sending .md requests directly to raw.githubusercontent.com.
Very Low Complexity

How it Works

nginx intercepts any request ending in .md and returns a 302 Found redirect to the matching file on GitHub's raw CDN. The URL mapping is direct: /docs/build/smart-contracts/overview.md → https://raw.githubusercontent.com/stellar/stellar-docs/main/docs/build/smart-contracts/overview.md. No build changes required. The browser follows the redirect and displays the raw source from GitHub.

Key limitation: The redirect uses the main branch, so the served markdown may not exactly match the deployed build if there are un-deployed commits on main. This also exposes GitHub's raw UI (no styling).

Approach 4
Nginx proxy_pass → GitHub Raw (transparent proxy)
nginx fetches the raw markdown from GitHub server-side and streams it to the client — no redirect, the original URL stays in the browser.
Medium Complexity

How it Works

Similar to Approach 3 but uses proxy_pass instead of return 302. When nginx receives a .md request, it silently fetches the file from GitHub Raw over HTTPS (server-to-server) and returns it to the client. The browser never sees a redirect — the .md URL remains in the address bar. This requires nginx to have outbound HTTPS access to raw.githubusercontent.com, which is available in Docker production environments.

The HTTPS proxy configuration is the tricky part: nginx needs proxy_ssl_server_name on to support SNI, and the resolver directive for DNS resolution inside the Docker container.

Files Changed

nginx/nginx.conf

✅ Pros

  • No build changes — nginx-only solution
  • URL stays intact in browser (no redirect)
  • Can add nginx-level caching (proxy_cache) for free
  • Client can't tell content came from GitHub

❌ Cons

  • More complex nginx config than Approach 3
  • nginx adds latency per request (proxy overhead)
  • Requires resolver directive in Docker (e.g., 8.8.8.8)
  • Same "main branch vs deployed" staleness issue
  • Same custom-slug limitation
  • Rate limiting from GitHub Raw (unauthenticated: 60 req/hr per IP)

Approach 5
Docusaurus contentLoaded Plugin (Slug-Aware)
A plugin with access to the fully-resolved Docusaurus content graph writes .md files to exactly the right paths, including pages with custom slug: frontmatter.
Medium Complexity

How it Works

Docusaurus exposes a loadContent + contentLoaded pair in the plugin API. By depending on the @docusaurus/plugin-content-docs loaded data (via allContent in postBuild), we can access every doc's resolved permalink alongside its source file path. This lets us write each .md file to exactly build/<resolved-permalink>.md, including documents where the frontmatter defines a custom slug:.

The plugin reads allContent['docusaurus-plugin-content-docs'] which contains the full document graph with permalink and source for every page.

+import fs from 'fs';
+import path from 'path';
+
+import type { LoadContext, Plugin } from "@docusaurus/types";
+
+interface DocRecord {
+  permalink: string; // e.g. /docs/build/smart-contracts/overview
+  source: string;    // e.g. @site/docs/build/smart-contracts/overview.md
+}
+
+export default async function mdExportFullPlugin(
+  context: LoadContext,
+): Promise<Plugin> {
+  return {
+    name: 'stellar-docs-md-export-full-plugin',
+
+    async postBuild({ outDir, plugins }) {
+      // Locate the loaded docs plugin instance to access its content
+      const docsPlugin = plugins.find(
+        (p) => p.name === 'docusaurus-plugin-content-docs',
+      );
+      if (!docsPlugin?.content) {
+        console.warn('md-export-full: could not find docs plugin content');
+        return;
+      }
+
+      // content.loadedVersions is an array; we only have "current" (unversioned)
+      const { loadedVersions } = docsPlugin.content as {
+        loadedVersions: Array<{ docs: DocRecord[] }>;
+      };
+
+      let count = 0;
+      for (const version of loadedVersions) {
+        for (const doc of version.docs) {
+          // source looks like "@site/docs/build/foo.md"
+          const sourcePath = doc.source
+            .replace('@site/', context.siteDir + '/')
+            .replace('@site\\', context.siteDir + '\\');
+
+          if (!fs.existsSync(sourcePath)) continue;
+
+          // Map permalink to build output path, preserving extension
+          const ext = path.extname(sourcePath); // .md or .mdx
+          const destPath = path.join(outDir, doc.permalink + ext);
+
+          fs.mkdirSync(path.dirname(destPath), { recursive: true });
+          fs.copyFileSync(sourcePath, destPath);
+          count++;
+        }
+      }
+
+      console.log(`✅ md-export-full: wrote ${count} files at canonical permalink paths`);
+    },
+  };
+}

Approach 6
Cloudflare Worker / CDN Edge Function
An edge function deployed to Cloudflare intercepts .md requests before they reach the origin server, fetches from GitHub, and serves raw Markdown inline.
High Complexity

How it Works

A Cloudflare Worker (or similar edge runtime) sits between the user and developers.stellar.org. When a *.md request arrives, the Worker intercepts it, maps the URL to the corresponding GitHub Raw path, fetches the content, and returns it with text/plain headers. All other requests pass through untouched. This approach requires Cloudflare to be in the CDN chain for the domain, which may or may not match the current deployment topology.

This is the most infrastructure-heavy option and introduces an external dependency (Cloudflare), but it requires zero changes to the Docusaurus build or Docker image.

  • Zero Docker image changes
  • Response cached at edge (low latency globally)
  • Can add auth, rate limiting, or transformation at edge

❌ Cons

  • Requires Cloudflare (or equivalent) in the CDN chain
  • Separate deployment pipeline for the Worker
  • Custom slug pages won't resolve (GitHub path mismatch)
  • Content lags behind unless pinned to a commit SHA
  • GitHub API rate limits (unauthenticated: 60/hr per IP)
  • Adds operational complexity (another thing to monitor)

+const GITHUB_RAW_BASE =
+  'https://raw.githubusercontent.com/stellar/stellar-docs/main';
+
+export default {
+  async fetch(request, env, ctx) {
+    const url = new URL(request.url);
+
+    if (/\.mdx?$/.test(url.pathname)) {
+      const rawUrl = `${GITHUB_RAW_BASE}${url.pathname}`;
+      const cacheKey = new Request(rawUrl);
+
+      // Check Workers Cache API first
+      const cache = caches.default;
+      let response = await cache.match(cacheKey);
+
+      if (!response) {
+        response = await fetch(rawUrl, {
+          headers: { 'User-Agent': 'stellar-docs-md-worker/1.0' },
+        });
+
+        if (!response.ok) {
+          return new Response('Not Found', { status: 404 });
+        }
+
+        // Cache for 5 minutes at the edge
+        response = new Response(response.body, {
+          headers: {
+            'Content-Type': 'text/plain; charset=utf-8',
+            'Cache-Control': 'public, max-age=300',
+          },
+        });
+        ctx.waitUntil(cache.put(cacheKey, response.clone()));
+      }
+
+      return response;
+    }
+
+    // Pass through all other requests to the origin
+    return fetch(request);
+  },
+};
