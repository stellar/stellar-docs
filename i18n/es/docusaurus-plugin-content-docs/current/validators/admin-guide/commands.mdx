El Núcleo Estelar puede ser controlado usando un CLI robusto.

<Alert>

Cubriremos una selección de los comandos _essential_ y la sintaxis aquí, pero el **mejor recurso** para utilizar la línea de comandos `stellar-core` se encuentra en el [stellar-core GitHub repo].

Además, mientras los comandos en esta página son comandos _CLI_, hay un conjunto adicional de [_HTTP_ endpoints commands] que proporcionan un control administrativo adicional sobre un nodo núcleo en ejecución.

</Alert>

## Obtener `--help` en cualquier lugar

La opción `--help` (aliases: `-h` o `-?`) puede especificarse en _cualquier lugar_ en la línea de comandos. Te mostrará el mensaje de ayuda para el comando correspondiente. Un ejemplo de uso es el siguiente:

```bash
stellar-core --help
stellar-core run --help
stellar-core --help new-db
stellar-core catchup --help
```

## Comandos esenciales

Para todos los comandos de stellar-core las opciones pueden _sólo_ colocadas después del comando.

### `new-db`

El comando **`new-db`** crea o restaura la base de datos local al editor de génesis.

#### Opciones de `new-db`

- `--minimal-for-in-memory-mode`: Restablece la base de datos especial usada sólo para el modo en memoria. (ver la bandera `--in-memory` en [opciones de `run`](#run-options))

### `ejecutar`

El comando **`run`** ejecutará el nodo `stellar-core`.

#### Opciones de `run`

- `--disable-bucket-gc`: Mantiene todo, incluso viejos, cubos en disco.
- `--metadata-output-stream <STREAM>`: Nombre de archivo o número de descriptor de archivo `fd:N` a los metadatos de stream a.
- `--in-memory`: Almacena el contador de trabajo en la memoria en lugar de la base de datos.
- `--wait-for-consensus`: Espere a escuchar de la red antes de votar, sólo para validar nodos.
- `--start-at-ledger <LEDGER>`: Comienza la ejecución en memoria con la repetición del número histórico del libro de notas.
- `--start-at-hash <HASH>`: Comienza la ejecución en memoria con la repetición desde el hash histórico de ledger.

### `catchup`

El comando **`catchup`** ejecutará un catchup de archivos de historia sin conectarse a la red.

#### Opciones de `catchup`

- `<DESTINATION_LEDGER/LEDGER_COUNT>`: (requerido) El libro de destino es cualquier número válido o `current` y el conteo de los libros de valores es cualquier número válido o `max`.
- `--archive <ARCHIVE-NAME>`: Nombre de archivo a ser usado para la recuperación. Usa `any` para seleccionar aleatoriamente.
- `--trusted-checkpoint-hashes <FILE-NAME>`: Obtener el hash del editor de destino de la salida confiable de `verify-checkpoints`.
- `--output-file <FILE-NAME>`: Archivo de salida.
- `--disable-bucket-gc`: Mantiene todo, incluso viejos, cubos en disco.
- `--extra-verification`: Verifica todos los archivos del archivo para el rango de recuperación.
- `--in-memory`: Almacena el contador de trabajo en la memoria en lugar de la base de datos.
- `--trusted-hash <HASH>`: Hash of the ledger to catchup.
- `--force-untrusted-catchup`: Forzar la recuperación no verificada.
- `--metadata-output-stream <STREAM>`: Nombre de archivo o número de descriptor de archivo `fd:N` a los metadatos de stream a.
- `--force-back`: Forzar estado del libro a un estado anterior, preservando datos históricos antiguos.
