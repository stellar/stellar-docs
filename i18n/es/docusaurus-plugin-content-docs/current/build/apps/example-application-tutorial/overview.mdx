En este tutorial, pasaremos por los pasos necesarios para construir una aplicación de pago básica en Testnet de Stellar. Después de este tutorial, deberías tener una buena comprensión de los conceptos estelares fundamentales y una base sólida para el desarrollo iterativo.

Para este tutorial, pasaremos por los pasos mientras construimos una aplicación de ejemplo que hemos llamado [BasicPay], que se utilizará para mostrar varias características.

:::caution

Aunque BasicPay es una aplicación completa en la red de Stellar, ha sido construido exclusivamente para mostrar la funcionalidad de Stellar para los propósitos educativos de este tutorial, no para ser copiado, pegado, y utilizado en Mainnet.

:::

## Configuración del proyecto

### Requisitos del proyecto

Para construir una aplicación estelar básica, necesitarás:

- Framework: estamos usando [SvelteKit] optando por JavaScript con escritura JSDoc. SvelteKit es bastante flexible y podría ser utilizado para muchos, pero lo estamos utilizando principalmente para sus capacidades de enrutamiento para minimizar este es un "tutorial de SvelteKit".
- Marco de interfaz: estamos usando [DaisyUI] para simplificar el uso de [Viento CSS].
- Una forma de interactuar con la red Stellar: estamos usando el [`js-stellar-sdk`], pero podría utilizar las solicitudes tradicionales de recuperación. El [`js-stellar-sdk`] también se utiliza para construir transacciones que se sometan a la red Stellar.
- Una forma de interactuar con el par de claves del usuario: estamos usando el SDK [`js-stellar-wallets], pero puedes optar por usar una cartera existente.

:::note

Mientras que estamos usando los componentes anteriores para construir nuestra aplicación, hemos hecho todo lo posible por escribir este tutorial de tal manera que la dependencia de cualquiera de estas cosas se minimiza. Idealmente, debería ser capaz de usar el código JavaScript que hemos escrito y conectarlo a cualquier otro framework que desee con un mínimo esfuerzo.

:::

Hemos hecho las siguientes opciones durante el desarrollo de BasicPay que puede que también necesite considerar a medida que siga aún:

- Hemos diseñado esta aplicación para el ordenador. En su mayoría, la aplicación responde a varios tamaños de pantalla, pero hemos _elegido_ para no salir de nuestro camino para priorizar la experiencia de usuario móvil.
- Hemos habilitado los temas "claros" y "oscuro" de DaisyUI por defecto, que deberían cambiar con las preferencias de su dispositivo. No hay ningún interruptor habilitado, sin embargo.
- Esto está escrito como una aplicación del cliente. En realidad, no se producen acciones en el lado del servidor. Si está construyendo una aplicación con un backend y un frontend, necesitaremos considerar cuidadosamente qué información vive donde, especialmente cuando se involucra la clave secreta del usuario.
- Estamos desplegando esto como una "aplicación de una sola página" estática con [Cloudflare Pages]. Sus propias decisiones de implementación tendrán un impacto en su configuración y proceso de compilación.
- Es probable que la aplicación no sea lo más rentable que podría ser. Tampoco está tan optimizada como podría ser. Hemos intentado encapsular las diversas funcionalidades de una manera que tiene sentido para el desarrollador leyendo el código, así que hay alguna duplicación de código y las cosas _podrían_ ser hechas de una manera "mejor".
- Nosotros _alguno_ manejamos errores, pero no tanto como desearías para una aplicación del mundo real. Si algo parece no funciona, y no estás viendo un error, abre tu consola de desarrollador y podrías averiguar qué ha ido mal.
- No hemos implementado _any_ pruebas automáticas. Probablemente querrá algunas para su aplicación.

:::note

Probablemente este tutorial se vea mejor como "_cerca_ ensivo." No vamos a guiarte por cada archivo de nuestra base de código, y los archivos que usamos para ilustrar conceptos en el tutorial pueden no estar _completamente_ presentes o explicados. Sin embargo, cubriremos lo básico y le señalaremos ejemplos más completos en el código base cuando sea aplicable.

:::

## Comenzando

Estos son los pasos que hemos tomado para empezar a construir BasicPay. Siéntase libre de inspirarse y personalizar estas direcciones como considere oportuno. Toda la [BasicPay Code] está abierta y disponible en GitHub para referencia.

:::note

### Install frameworks

Lo primero que necesitaremos hacer es crear una aplicación SvelteKit, usando `npm`, estamos usando v18.x de nodejs.

```bash
npm create svelte@latest my-basic-payment-app
```

Esto le guiará a través del proceso de creación de SvelteKit, preguntándole por las distintas opciones. Hemos elegido las siguientes opciones:

- **¿Cuál plantilla de aplicación Svelte?** Proyecto Skeleton
- **¿Está comprobando el tipo con TypeScript?** Sí, usando JavaScript con comentarios JSDoc
- **Selecciona opciones adicionales** Añade ESLint para el linting de código; añade Prettier para el formato de código

Después de este proceso, el andamio para tu aplicación vivirá dentro del directorio `my-basic-payment-app`. Puedes `cd` en ese directorio y añadir algunas dependencias de la interfaz de usuario.

```bash npm2yarn
cd my-basic-payment-app
npm install --save-dev svelte-preprocess tailwindcss autoprefixer postcss @sveltejs/adapter-static \
    @tailwindcss/typography \
    daisyui svelte-feather-icons
```

Antes de configurar nada, necesitaremos generar nuestros archivos `tailwind.config.js` y `postcss.config.js`.

```bash
npx tailwindcss init -p
```

Ahora, necesitaremos un poco de configuración para hacer que todos esos componentes funcionen juntos. Primero, modifica tu archivo `svelte.config.js`:

```js title="/svelte.config.js"
// resaltado inicio
importar preproceso desde "svelte-preprocess";
import adapter from "@sveltejs/adapter-static";
// resaltado

/** @type {import('@sveltejs/kit'). onfig} */
const config = {
  kit: {
    // Nota: Tu configuración `dapter` puede necesitar personalizaciones dependiendo
    // sobre cómo estás construyendo e implementando tu aplicación.
    // Resaltado de
    adaptador: adapter({
      fallback: "index. tml",
    }),
    // resaltado
  },
  // resaltar-iniciar
  preprocesos: [
    preprocesos({
      postcss: true,
    }),
  ],
  // resaltado
};

configuración predeterminada de exportación;
```

A continuación, puede configurar el archivo `tailwind.config.js`.

1. Importa los plugins `daisyui` y `typography`
2. Configure nuestras rutas de contenido (puede que necesite modificar estos valores dependiendo de la estructura de su proyecto)
3. Añade el complemento `daisyui` **después** de cualquier complemento oficial `@tailwindcs` (solo `typography` en nuestro ejemplo)

```js title="/tailwind.config.js"
// resaltado-inicio
const daisyui = require("daisyui");
const typography = require("@tailwindcss/typography");
// resaltado

/** @type {import('tailwindcss'). onfig} */
exportar por defecto {
  contenido: [
    // resaltar-iniciar
    ". src/routes/**/*.{html,js,svelte,ts}",
    "./src/routes/**/*.{html,js,svelte,ts}",
    "./src/lib/components/**/*. html,js,svelte,ts}",
    // resaltado
  ],
  // resaltado next-line
  plugins: [typography, daisyui],
};
```

Añade tus directivas de viento a medida al archivo CSS principal de tu aplicación.

```css title="/src/app.postcss"
@tailwind base;
@tailwind componentes;
@tailwind utilidades;
```

A continuación, importa el archivo CSS en el diseño base de SvelteKit (puede que necesites crear este archivo).

```html title="/src/routes/+layout.svelte"
<script>
  importar "../app.postcss";
</script>

<slot />
```

También creamos un archivo `/src/routes/+layout.js` para configurar nuestra aplicación como _only_ cliente. Esto significa que la aplicación se entregará al cliente como HTML no renderizado y JavaScript.

```js title="/src/routes/+layout.js"
// Desactivar pre-renderizado de páginas durante el tiempo de compilación
export const )[video] ender = false;
// Deshabilitar renderizado del lado del servidor
export const ssr = false;
```

¡Su proyecto SvelteKit ahora está configurado y listo para ejecutarse!

```bash npm2yarn
npm run dev
```

### Dependencias estelares

Para trabajar con la red Stellar, estructuras de datos y pares de claves almacenados localmente, vamos a instalar y configurar algunas dependencias más.

```bash npm2yarn
# Stellar SDKs
npm install --save-dev stellar-sdk @stellar/wallet-sdk
# Necesitaremos algunos polyfills para que las cosas estén disponibles en el lado del cliente
npm install --save-dev @esbuild-plugins/node-globals-polyfill @esbuild-plugins/node-modules-polyfill \
    ruta @rollup/plugin-inject buffer svelte-local-storage-store uid
```

Utilizaremos un archivo `window.js` para inyectar Buffer en nuestro código del lado del cliente, ya que eso es requerido por algunas partes del `stellar-sdk`.

```js title="/src/lib/window.js"
import { browser } from "$app/environment";
import { Buffer } from "buffer";

if (browser) {
  window. uffer = Buffer;
} else {
  globalThis.Buffer = Buffer;
  globalThis.window = {};
}
export default globalThis;
```

La "inyección" real tiene lugar en nuestro archivo `vite.config.js`.

```js title="/vite.config.js"
importar { sveltekit } desde "@sveltejs/kit/vite";
importar { defineConfig } desde "vite";
importar { NodeGlobalsPolyfillPlugin } desde "@esbuild-plugins/node-globals-polyfill";
importar inyección desde "@rollup/plugin-inject";
importar ruta desde "ruta";

exportar defineConfig({
  plugins: [sveltekit()],
  optimizeDeps: {
    esbuildOptions: {
      define: {
        global: "globalThis",
      },
      plugins: [
        NodeGlobalsPolyfillPlugin({
          buffer: true,
        }),
      ],
    },
  },
  build: {
    rollupOptions: {
      plugins: [
        inject({
          window: ruta. esolve("src/lib/window. s"),
        }),
      ],
    },
  },
  ssr: {
    noExternal: ["@stellar/wallet-sdk", "@albedo-link/intent"],
  },
});
```
