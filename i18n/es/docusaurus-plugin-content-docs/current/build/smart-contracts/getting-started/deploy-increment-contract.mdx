## Despliegue en dos pasos

Vale la pena saber que `deploy` es en realidad un proceso de dos pasos.

1. \*_Sube los bytes del contrato a la red. _ Soroban se refiere actualmente a esto como _instalando_ el contrato—desde la perspectiva del propio blockchain, esta es una metáfora razonable. Esto sube los bytes del contrato a la red, indexándolo por su hash. Este código de contrato ahora puede ser referenciado por varios contratos, lo que significa que tendrían exactamente el mismo _comportamiento_ pero el mismo estado de almacenamiento separado.

2. **Instanciar el contrato.** Esto crea realmente lo que probablemente pienses como un Contrato Inteligente. Hace un nuevo ID del contrato, y lo asocia con los bytes del contrato que se subieron en el paso anterior.

Puede ejecutar estos dos pasos por separado. Intentémoslo con el contrato Incrementar:

<Tabs groupId="platform" defaultValue={getPlatform()}>

<TabItem value="unix" label="macOS/Linux">

```sh
stellar contract install \
  --network testnet \
  --source alice \
  --wasm target/wasm32-unknown-unknown/release/soroban_increment_contract.wasm
```

</TabItem>

<TabItem value="windows" label="Windows (PowerShell)">

```powershell
stellar contract install `
  --network testnet `
  --source alice `
  --wasm target/wasm32-unknown-unknown/release/soroban_increment_contract.wasm
```

</TabItem>

</Tabs>

Esto devuelve el hash de los bytes de Wasm, como `6ddb28e0980f643bb97350f7e3bacb0ff1fe74d846c6d4f2c625e766210fbb5b`. Ahora puedes usar `--wasm-hash` con `deploy` en lugar de `--wasm`:

<Tabs groupId="platform" defaultValue={getPlatform()}>

<TabItem value="unix" label="macOS/Linux">

```sh
despliegue de contrato estelar \
  --wasm-hash 6ddb28e0980f643bb97350f7e3bacb0ff1fe74d846c6d4f2c625e766210fbb5b \
  --source alice \
  --network testnet
```

</TabItem>

<TabItem value="windows" label="Windows (PowerShell)">

```powershell
despliegue de contrato estelar `
  --wasm-hash 6ddb28e0980f643bb97350f7e3bacb0ff1fe74d846c6d4f2c625e766210fbb5b `
  --source alice `
  --network testnet
```

</TabItem>

</Tabs>

Este comando devolverá el ID del contrato (por ejemplo, `CACDYF3CYMJEJTIVFESQYZTN67GO2R5D5IUABTCUG3HXQSRXCSOROBAN`), y puede usarlo para invocar el contrato como lo hicimos en ejemplos anteriores.

<Tabs groupId="platform" defaultValue={getPlatform()}>

<TabItem value="unix" label="macOS/Linux">

```sh
stellar contract invoke \
  --id CACDYF3CYMJEJTIVFESQYZTN67GO2R5D5IUABTCUG3HXQSRXCSOROBAN \
  --source alice \
  --network testnet \
  -- \
  incremento
```

</TabItem>

<TabItem value="windows" label="Windows (PowerShell)">

```powershell
contrato estelar invoca `
  --id CACDYF3CYMJEJTIVFESQYZTN67GO2R5D5IUABTCUG3HXQSRXCSOROBAN `
  --source alice `
  --network testnet `
  -- `
  incremento
```

</TabItem>

</Tabs>

Debería ver la siguiente salida:

```sh
1
```

Ejecute un par de veces más para ver el cambio de cuenta.

## Ejecutar su propia red/nodo

A veces necesitarás ejecutar tu propio nodo:

- ¡Aplicaciones de producción! Stellar mantiene los nodos RPC públicos de prueba para Testnet y Futurenet, pero no para Mainnet. En su lugar, necesitará ejecutar su propio nodo y apuntar su aplicación a eso. Si quieres usar una plataforma de software-as-a-service para esto, [varios proveedores](../../data/rpc/rpc-providers.mdx) están disponibles.
- Cuando necesita una red que difiera de la versión desplegada a Testnet.

El equipo RPC mantiene contenedores Docker que lo hacen lo más sencillo posible. Vea la referencia [RPC](../../../data/rpc/admin-guide.mdx) para más detalles.
