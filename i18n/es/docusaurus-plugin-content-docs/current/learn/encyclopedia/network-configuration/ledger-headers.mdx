Cada libro tiene una cabecera que hace referencia a los datos en ese libro y en el contador anterior. Estas referencias son hash criptográficos del contenido que se comportan como punteros en estructuras de datos típicas pero con mayores garantías de seguridad. Piense en una cadena histórica de contadores como una lista vinculada de cabeceras de contadores:

```mermaid
estadoDiagrama
    dirección LR
    Génesis --> LedgerHeader_1
    LedgerHeader_1 --> ...
    ... --> LedgerHeader_n
```

El libro principal de génesis tiene un número de secuencia de 1. El libro de valores directamente siguiendo un libro con el número de secuencia n tiene un número de secuencia de n+1.

## Campos de cabecera Ledger

### Versión

La versión de protocolo de este notario.

### Hedger anterior

Hash del contador anterior.

### Valor SCP

Durante el consenso, todos los nodos de validación en la red ejecutan SCP y están de acuerdo en un valor en particular. que es un conjunto de transacciones que se aplicarán a un notario. Este valor se almacena aquí y en los siguientes tres campos (conjunto de transacciones hash, tiempo de cierre y actualizaciones).

### Transacción con hash

Hash de la transacción establecida aplicada al contador anterior.

### Hora de cierre

La hora de cierre es una marca de tiempo UNIX que indica cuándo se cierra el registro. Su precisión depende del reloj del sistema del validador proponiendo el bloque. En consecuencia, SCP puede confirmar un tiempo de cierre que marca unos segundos atrás o hasta 60 segundos por delante. Es estrictamente monótono – garantizado ser mayor que el tiempo de cierre de un contador anterior.

### Mejoras

Cómo la red ajusta los valores generales (como la tarifa base) y acuerda cambios en toda la red (como cambiar a una nueva versión de protocolo). Este campo está normalmente vacío. Cuando hay una actualización en toda la red, el SDF informará y ayudará a coordinar a los participantes usando el canal #validators en Dev Discord y el Stellar Validators Grupo de Google.

### La transacción establece el hash del resultado

Hash de los resultados de la aplicación del conjunto de transacciones. Estos datos no son necesarios para validar los resultados de las transacciones. Sin embargo, hace que sea más fácil para las entidades validar el resultado de una transacción determinada sin tener que aplicar la transacción establecida al contador anterior.

### Hash de la lista de cubos

Hash de todos los objetos en este contador. La estructura de datos que contiene todos los objetos se llama la lista de cubetas.

### Secuencia de Ledger

El número de secuencia de este contador.

### Monedas totales

Número total de lumens en existencia.

### Piscina

Número de lumens que han sido pagados en honorarios. Tenga en cuenta que esto se denomina en lumens, aunque el campo de tarifa de una transacción está en golpes.

### Secuencia de inflación

Número de veces que se ha mantenido la inflación. Nota: la operación de inflación fue obsoleta cuando los validadores votaron para actualizar la red al Protocolo 12 en 10/28/2019. Por lo tanto, la inflación ya no funciona, por lo que esta secuencia ya no cambia.

### Piscina de ID

El último ID global utilizado. Estos IDs se utilizan para generar objetos.

### Número máximo de transacciones

El número máximo de validadores de operaciones han acordado procesar en un contador determinado. Si se presentan más transacciones que este número, la red entrará en el modo de sobretensión. Para obtener más información acerca de las estrategias de precios por sobretensiones y tarifas, consulte nuestra [Sección de Tarifas](../../fundamentals/fees-resource-limits-metering.mdx).

### Tarifa base

El cargo que la red cobra por operación en una transacción. Calculado en tropas. Consulte la [Sección de Tarifas](../../fundamentals/fees-resource-limits-metering.mdx) para más información.

### Reserva base

La reserva que la red utiliza al calcular el saldo mínimo de una cuenta.

### Omitir lista

Hashes de bordes en el pasado. Le permite saltar a tiempo en la cadena del libro sin retroceder en el libro de contadores. En la lista de saltos se almacenan cuatro hash del libro de notas. Cada franja contiene el libro mayor que es mod de 50 5000 50000 o 500000 dependiendo del índice skipList[0] mod(50), saltar lista[1] mod(5000), etc.
