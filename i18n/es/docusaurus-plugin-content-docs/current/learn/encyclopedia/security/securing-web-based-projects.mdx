Cualquier aplicación que gestione criptomonedas es un objetivo frecuente de actores maliciosos y necesita seguir las mejores prácticas de seguridad. La siguiente lista ofrece orientación sobre las vulnerabilidades más comunes. Sin embargo, incluso si usted sigue todos los consejos, la seguridad no está garantizada. La seguridad web y los actores maliciosos están evolucionando constantemente, por lo que es bueno mantener una cantidad saludable de paranoia.

## SSL/TLS

Asegúrese de que TLS está activado. Redirigir HTTP a HTTPS cuando sea necesario para asegurarse de que el hombre en los ataques del Medio no puede ocurrir y que los datos sensibles se transfieren de forma segura entre el cliente y el navegador. Habilite TLS y obtenga un certificado SSL de forma gratuita en [LetsEncrypt](https://letsencrypt.org/getting-started/).

Si no tiene habilitado SSL/TLS, detenga todo y haga esto primero.

## Encabezados de la política de seguridad de contenido)

Las cabeceras CSP le indican al navegador de dónde puede descargar recursos estáticos. Por ejemplo, si astralwallet.io y solicita un archivo JavaScript de myevilsite.com, su navegador lo bloqueará a menos que se incluya en la lista blanca con cabeceras CSP. Puedes leer sobre cómo implementar cabeceras CSP [here](https://developer.mozilla.org/en-US/docs/Web/HTTP/CSP).

La mayoría de los frameworks web tienen un archivo de configuración o extensiones para especificar su política CSP, y los encabezados se generan automáticamente para usted. Por ejemplo, vea [Helmet](https://www.npmjs.com/package/helmet) para Node.js. Esto habría evitado el [Hackwallet Blackwallet Hack](https://www.ccn.com/yet-another-crypto-wallet-hack-causes-users-lose-400000/).

## Encabezados HTTP estrictos de transporte y seguridad

Esta es una cabecera HTTP que le dice al navegador que todas las conexiones futuras a un sitio en particular deben usar HTTPS. Para implementar esto, añade la [header](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Strict-Transport-Security) a tu sitio web. Algunos frameworks web (como [Django](https://docs.djangoproject.com/en/2.0/topics/security/#ssl-https)) tienen esto incorporado. Esto habría evitado el [Hack DNS de MyEtherWallet](https://bitcoinmagazine.com/articles/popular-ether-wallet-mew-hijacked-dns-attack/).

## Almacenando datos sensibles

Idealmente, no es necesario almacenar muchos datos confidenciales. Si es necesario, asegúrese de andar con cuidado. Hay muchas estrategias para almacenar datos sensibles:

- Asegurarse de que los datos sensibles sean cifrados usando un cifrado comprobado como AES-256 y almacenados separadamente de los datos de la aplicación. Recoger siempre el modo AEAD.
- Cualquier comunicación entre el servidor de aplicaciones y el servidor secreto debe estar en una red privada y/o autenticada a través de HMAC. Tu estrategia de cifrado cambiará en función de si vas a enviar el algoritmo de cifrado a través del cable varias veces.
- Haga una copia de seguridad de las claves de cifrado que pueda usar sin conexión y almacenarlas sólo en la memoria de su aplicación.
- Consulte a un buen criptografista y lea las mejores prácticas. Consulte la documentación de su framework web favorito.
- Rolear tu propia criptomoneda es una mala idea. Usar siempre bibliotecas probadas como [NaCI](https://en.wikipedia.org/wiki/NaCl_(software)).

## Monitoreo

Los atacantes a menudo necesitan pasar tiempo explorando su sitio web para un comportamiento inesperado o ignorado. Examinar los troncos de forma defensiva puede ayudarte a descubrir lo que están tratando de conseguir. Puede al menos bloquear su IP o bloquear automáticamente basándose en un comportamiento sospechoso.

También vale la pena configurar un informe de error (como [Sentry](https://sentry.io/welcome/)). A menudo, la gente activa errores extraños cuando intenta hackear las cosas.

## Debilidades de autenticación

Debe construir su autenticación de forma segura si tiene inicio de sesión para los usuarios. La mejor manera de hacerlo es utilizar algo fuera de la barrera. Tanto Ruby on Rails como Django tienen esquemas de autenticación integrados y robustos.

Muchas implementaciones de token web JSON están mal hechas, así que asegúrese de que la biblioteca que utiliza está auditada.

Las contraseñas de hash con un esquema comprobado en el tiempo son buenas. Y el Hashing de globos también vale la pena mirar.

Preferimos fuertemente 2FA y requerimos U2F o [TOTP](https://tools.ietf.org/html/rfc6238) 2FA para acciones sensibles. 2FA es importante ya que las cuentas de correo electrónico normalmente no son muy seguras. Tener un segundo factor de autenticación asegura que los usuarios que accidentalmente permanezcan conectados o tengan su contraseña adivinada todavía estén protegidos.

Por último, requieren contraseñas fuertes. Las contraseñas comunes y cortas pueden ser brutalmente forzadas. Dropbox tiene una gran [herramienta de código abierto](https://blogs.dropbox.com/tech/2012/04/zxcvbn-realistic-password-strength-estimation/) que recopila la fuerza de la contraseña bastante rápidamente, haciéndola utilizable para interacciones de usuario.

## Ataques de denegación de servicio (DOS)

Los ataques DOS normalmente se completan sobrecargando sus servidores web con tráfico. Para mitigar este riesgo, la tasa limite el tráfico de las huellas digitales de los navegadores y las IP. A veces la gente usará proxies para eludir la limitación de la tasa de IP. Al final, los actores maliciosos siempre pueden encontrar formas de falsificar su identidad, por lo que la forma más segura de bloquear ataques DOS es implementar comprobaciones de trabajo en tu cliente o usar un servicio administrado como [Cloudflare](https://www. loudflare.com/ddos/).

## Bloquear puertos sin usar

Los atacantes a menudo escanean sus puertos para ver si usted era negligente y dejó cualquier abierto. Servicios como Heroku hacen esto por ti- [lee sobre cómo activar esto en AWS](https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/authorizing-access-to-an-instance.html).

## Phishing e ingeniería social

Los ataques de suplantación de identidad frustrarán cualquier infraestructura de seguridad bien formada. Tenga políticas claras publicadas en su sitio web y árticelas a los usuarios cuando se registren (usted nunca pedirá su contraseña, etc.). Firma mensajes a tus usuarios y pide a los usuarios que comprueben el dominio del sitio web en el que se encuentran.

## Escanea tu sitio web y bibliotecas en busca de vulnerabilidades

Utilice una herramienta como [Snyk](https://snyk.io/) para buscar vulnerabilidades en las librerías de clientes de terceros. Asegúrese de mantener actualizadas sus bibliotecas de terceros. A menudo, las actualizaciones son activadas por exploits de seguridad. Puedes usar [Mozilla Observatory](https://observatory.mozilla.org/) para comprobar también tu seguridad HTTP.

## Protección contra la falsificación de la solicitud en varios sitios, inyecciones SQL

La mayoría de los frameworks web y móviles modernos manejan tanto la protección CSRF como las inyecciones SQL. Asegúrese de que la protección CSRF está habilitada y de que está utilizando una base de datos ORM en lugar de ejecutar SQL crudo basado en la entrada del usuario. Por ejemplo, vea lo que [Ruby on Rails documentation](http://guides.rubyonrails.org/security.html#sql-injection) dice sobre inyecciones SQL.
