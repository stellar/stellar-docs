Soroban-RPC aceptará peticiones HTTP POST usando la especificación [JSON-RPC 2.0]. Los errores se devuelven a través del [objeto de error jsonrpc] siempre que sea posible (y tiene sentido).

Para la producción y otras instancias públicamente accesibles, el punto final JSON-RPC debe ser servido sobre SSL en el puerto 443, donde sea posible, para la seguridad y facilidad de uso. Though, soroban-rpc no termina ssl por sí mismo, así que necesitará un balance-carga u otro servicio para terminar SSL para él.

Para interactuar con soroban-rpc desde dentro de una aplicación JavaScript, utilice el paquete \[SDK de JavaScript. que proporciona una interfaz conveniente para los métodos RPC dentro de su módulo `SorobanRpc`.

Cuando XDR se pasa como parámetro o se regresa, siempre es una cadena codificada usando base64 estándar.

:::info

Tenga en cuenta que la estructura de parámetros debe contener parámetros nombrados como un objeto por nombre, y no como argumentos posicionales como una matriz por posición. Los argumentos posicionales como un array por posición serán obsoletos en futuras versiones RPC.

:::

## Especificación Open-RPC

Soroban-RPC proporciona un documento [OpenRPC] de especificación que puede ser utilizado para simular, construir y/o validar implementaciones de software tanto del servidor como del cliente. Este documento se utiliza para generar todas nuestras páginas [methods] de documentación. Puede ver el documento completo [documento de especificación aquí]. Además, puede experimentar con este documento específico en el [OpenRPC Playground].
