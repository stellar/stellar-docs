El Soroban-RPC puede describirse como una "pasarela de red en vivo para Soroban". Proporciona información que la red tiene actualmente en su vista (es decir, el estado actual). También tiene la capacidad de enviar una transacción a la red y consultar el estado de las transacciones enviadas previamente. Está pensado para ser simple, mínimo, escalable y familiar para los desarrolladores de blockchain de otros ecosistemas.

Soroban-RPC debe proporcionar todos los datos básicos que un desarrollador dapp necesitaría, siempre que sean:

- Sólo interesado en los datos del estado actual, o están dispuestos a ingerir eventos en su propia infraestructura para apoyar consultas de informes y analíticos.
  - Aviso: Soroban-RPC debe proporcionar suficiente retención de datos para apoyar la ingestión confiable de eventos.
- Sólo interesado en construir y someter las transacciones de Soroban (no Stellar Vanilla).

Soroban-RPC debería apoyar al desarrollador desde las pruebas locales (a través de la imagen de inicio rápido), hasta las implementaciones de producción.

- Esto implica que debería ser fácil de desplegar y fácil de mantener; con bajo costo y poco "administrador" necesario.
- El desarrollador debería ser capaz de ejecutar simplemente la imagen docker de inicio rápido, y estén listos para atender solicitudes sin necesidad de instalar o mantener una infraestructura dependiente.

## Antiobjetivos

- Soroban-RPC no es un reemplazo Horizon. Horizon es más adecuado para consultas de informes y análisis históricos. Horizon se ocupa más de los datos estelares "clásicos".
- Soroban-RPC no debe depender de Horizon. Horizon es costoso y difícil de correr, así que si Soroban-RPC dependiera de Horizon, heredaría eso.
- Soroban-RPC también debería proporcionar una base de API para que los proveedores de infraestructura implementen, pero no necesariamente sea una solución fuera de control para ellos.
- Soroban-RPC no es una herramienta de información/análisis y no proporciona una profunda consulta de datos históricos. No proporciona ninguna agregación de datos, compilaciones o solicitudes de coordinación multiservidor.
- Soroban-RPC no apunta a la latencia ultra baja. La latencia debería ser lo suficientemente baja como para crear y presentar transacciones exitosas de soroban, pero apoyar a los comerciantes de alta frecuencia no es el objetivo.
